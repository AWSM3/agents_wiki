# AGENTS.md — универсальный “briefing” для coding‑агентов (template)

## 1) Project overview (1–2 абзаца)

Опишите:

- что это за репозиторий (сервисы/приложения)
- ключевые ограничения (security, performance, style)
- что считается “готово” (DoD)

## 2) Quick commands

> Заполните реальные команды проекта.

- Install: `…`
- Lint: `…`
- Test: `…`
- Build: `…`

## 3) Working agreement (как работать с агентом)

- Сначала **Plan** (шаги + файлы + риски + тест‑план).
- Большие изменения — только через **STOP‑POINT** (ждать подтверждение).
- Делать **минимальный diff**: менять только относящееся к задаче.
- Всегда выполнять **Verify** (tests/build/static) перед “готово”.

## 4) Code style & architecture

- ссылки на архитектурные документы/ADR
- соглашения по именованию/структуре модулей
- запреты на “оверинжиниринг”

## 5) Safety / Data policy (критично)

- Никогда не вставлять секреты (`.env`, ключи, токены, kubeconfig).
- Не раскрывать доменные модели/уникальные алгоритмы во внешний контур.
- Если контекст D2/D3 — работать **локально/on‑prem**.

## 6) “Do not touch” zones

Перечислите папки/модули, которые нельзя менять без code owner’а:

- auth/crypto
- CI/CD
- database migrations

## 7) Output expectations

При выдаче результата агент должен:

- объяснить, что изменил и почему
- перечислить проверки и команды для запуска
- указать риски и rollback‑план
