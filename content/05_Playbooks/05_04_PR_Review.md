# 5.4. Playbook: анализ PR / ревью / поиск багов

- **Owner**: TL / QA (TBD)
- **Статус**: Draft
- **Last reviewed**: 2026-01-15
- **Уровень**: Basic
- **Для ролей**: Dev / TL / QA

## Когда применять

- как “второй рецензент” перед merge
- при подозрении на регрессию/безопасностный риск

## Input

- diff/PR description
- Spec/AC (если есть)
- зоны риска проекта (auth/db/crypto/CI)

## Output

- отчёт по рубрике: correctness / security / performance / style / tests
- список конкретных замечаний и severity (S0–S3)

## Process

1. Попросить агента кратко пересказать намерение PR (что меняется).
2. Прогнать рубрику и отметить severity.
3. Отдельно: “что может сломаться” + как проверить.

## Safety

- агент не заменяет code owners ревью
- S2–S3 замечания требуют ручной проверки

## Template prompt

> Проанализируй PR по рубрике: correctness/security/perf/style/tests.  
> Для каждого замечания укажи severity S0–S3 и предложи конкретный фикс/проверку.

## Done criteria

- основные риски/регрессии выявлены
- проверки/тест‑план понятны

## Pitfalls

- “общие слова” без привязки к diff
