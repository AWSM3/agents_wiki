# 1.3. Политика “что можно/нельзя отправлять в LLM” (коротко)

- **Last reviewed**: 2026-01-15
- **Уровень**: Basic

## Зачем эта страница

У каждого должна быть “карточка безопасности” на 1 страницу: что допустимо давать агенту, а что — нет.

## Красная зона (нельзя)

Нельзя отправлять во внешний контур (и часто нельзя в LLM вообще):

- **секреты**: ключи/токены/сертификаты/пароли, `.env`, kubeconfig, приватные ключи
- **уникальные алгоритмы/ноу‑хау** (то, что даёт конкурентное преимущество)
- **доменные модели** (уникальные сущности, бизнес‑схемы, правила)
- **схемы БД/таблиц/полей/индексы**, которые формируют вектор атаки
- **prod‑логи** с PII/фин.данными/секретами (см. классы чувствительности D2/D3)

## Жёлтая зона (осторожно)

Можно, но только после редактирования/обобщения:

- маленькие фрагменты кода без доменных имён и без “сквозной логики”
- логи/трейсы с редактированными идентификаторами, без hostnames/VPN‑домена
- SQL — только на обезличенной схеме (table_A/col_x), без реальных индексов

## Зелёная зона (можно)

- общие вопросы по языкам/паттернам/инструментам
- синтетические примеры и псевдокод
- публичные библиотеки/документация
- шаблоны документов (Spec/ADR/TestPlan), без чувствительного контекста

## Правило по умолчанию

**Если сомневаешься — делай локально** и/или используй on‑prem контур. См. [8.1 Когда локально](../Локальные_Модели/Когда_Локально.md).
