# 1.4. Быстрый старт: 30 минут до первого полезного результата

- **Last reviewed**: 2026-01-15
- **Уровень**: Basic

## Цель

За 30 минут получить "первую пользу" **по стандарту**: маленькая правка, понятный diff, проверки, оформленный результат.

## Мини‑сценарий (рекомендуется)

Выберите задачу уровня "low risk":

- добавить unit‑тест на существующую функцию + минимальный рефакторинг
- поправить обработку ошибки (без изменения публичного API)
- улучшить читаемость/логирование (без изменений бизнес‑логики)

## Практикум (30 минут)

1. **Включите корректные privacy/retention настройки** для выбранного инструмента (см. [6.6](../Безопасность_Соответствие/Приватность_Хранение_По_Инструментам.md)).
2. **Подключите инструкции проекта** (если есть): AGENTS.md / rules / команды.
3. **Соберите Context Pack** (в чате или файлом):
   - цель задачи
   - ограничения (не трогать: auth/db/crypto/CI и т.п.)
   - какие файлы можно менять
   - критерии приёмки (AC)
4. **Попросите сначала план**:
   - шаги
   - список файлов
   - риски регрессии
   - тест‑план
5. **STOP‑POINT**: подтвердите план перед внесением изменений.
6. **Внесите изменения** маленьким diff'ом.
7. **Verify**: прогоните тесты/линтер/сборку (что применимо).
8. **Сверьте с DoD** и зафиксируйте результат (MR/PR описание по шаблону).

## Шаблон запроса (копипаст)

> Цель: …  
> Ограничения: без новых зависимостей; минимальный diff; не трогать публичный API; не менять папки: …  
> Контекст: файлы: … (укажу/приложу)  
> Acceptance criteria: …  
> Сначала дай: 1) план, 2) список файлов, 3) риски, 4) тест‑план.  
> После плана дождись моего "OK".

## Риски и анти‑паттерны

- "Сделай просто" без контекста → низкая предсказуемость.
- Массовые правки без плана → высокий риск регрессий.
- "Готово" без проверок → ложные победы.
