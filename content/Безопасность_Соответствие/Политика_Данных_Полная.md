# 6.1. Политика данных и NDA (полная версия)

- **Last reviewed**: 2026-01-15
- **Уровень**: Basic → Advanced

## Зачем эта страница

Дать единый ответ "что можно/нельзя" и убрать самодеятельность. Это базовый слой безопасности для любых AI‑агентов.

## Классификация данных (пример)

Можно использовать 4 класса (адаптируйте под вашу орг‑модель):

- **Public** (D0): публичные данные, можно в облако.
- **Internal** (D1): внутренние данные без секретов/PII, можно в облако с ограничениями.
- **Sensitive** (D2): чувствительные данные, только on‑prem/локально.
- **Restricted/Secret** (D3): секреты/строго ограниченные, нельзя в LLM (или только строго изолированный контур).

## Запрещено передавать во внешний контур (минимум)

- секреты (ключи/пароли/серты/токены), `.env`, kubeconfig
- доменные модели/уникальные алгоритмы/ноу‑хау
- схемы БД/индексы/поля (если дают вектор атаки)
- прод‑логи с PII/финансовыми данными

## Разрешено (при соблюдении правил)

- псевдокод/обобщённые примеры
- синтетические данные
- "минимальные фрагменты" кода после редактирования/обезличивания
- внутренние доки без чувствительных деталей (по вашей политике)

## Политика маршрутизации (куда отправляем)

- **D0/D1**: допускается SaaS (при корректных privacy/retention настройках).
- **D2**: только on‑prem/локально (или через корпоративный шлюз с DLP).
- **D3**: запрещено; исключения — только через formal‑process и изоляцию.

## Обязательные меры (минимум)

- запрет вставки секретов в контекст
- редактирование логов/кода перед передачей (см. [6.4](Безопасный_Промптинг_Редактирование.md))
- уровни автономности и STOP‑POINT'ы (см. [10.4](../Приложения/Руководства_и_Критерии_оценки/Уровни_автономности_A0_A4.md))
- аудит и code review для зон риска (auth/db/crypto/CI)
