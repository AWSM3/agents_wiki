# 6.4. Safe prompting & redaction (Basic)

- **Last reviewed**: 2026-01-15
- **Уровень**: Basic

## Зачем эта страница

Дать простой чек‑лист: как давать контекст агенту так, чтобы не утекли секреты/PII и чтобы результат был качественным.

## Красные флаги (никогда не вставлять)

- `.env`, kubeconfigs, токены, приватные ключи
- секреты из CI/CD (variables, secrets)
- реальные hostnames/VPN‑домены/внутренние URL

## Редакция контекста (перед тем как вставить код/логи)

- **Обезличить домен**:
  - Customer → EntityA
  - Order → EntityB
- **Обезличить схему**:
  - `orders` → `table_A`
  - `customer_id` → `col_x`
- **Удалить идентификаторы**:
  - request ids, user ids (заменить на `id_123`)
- **Скрыть сеть**:
  - убрать внутренние IP/хосты/эндпоинты

## Шаблон запроса к LLM (копипаст)

> Даю обезличенный пример. В нём нет секретов/PII.  
> Не проси реальные схемы/ключи/внутренние hostnames.  
> Сначала дай: 1) план, 2) патч/шаги, 3) проверки, 4) риски.

## Анти‑паттерны

- вставить “кусок .env” “на минутку”
- дать реальную схему БД “чтобы агенту проще”
- просить “сделай всё сам” без ограничений
