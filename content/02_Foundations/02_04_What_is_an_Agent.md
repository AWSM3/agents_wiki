# 2.4. Что такое “агент”: планирование, итерации, память, инструменты

## Зачем эта страница

Чтобы понимать, почему агент “не равен” чат‑боту: он не только отвечает, но и **планирует действия** и **выполняет их через инструменты**.

## Цикл агента (упрощённо)

1. **Цель** (что должно измениться)
2. **План** (шаги, файлы, риски, тест‑план)
3. **Действия** (tools: поиск/чтение/патч/команды)
4. **Проверка** (tests/build/static, сверка с AC)
5. **Итерация** (если проверки не прошли)

## Виды “агентности” на практике

- **IDE‑агент (синхронный)**: работает в вашей IDE, быстро итеративно правит код.
- **Terminal‑агент**: удобен для многошаговых задач (миграции, CI, сборки, скрипты).
- **PR/CI‑агент**: работает в изоляции и приносит MR/отчёт.
- **Долгоживущий автономный агент**: может работать часами/днями — требует самых жёстких guardrails.

## Риски агента

- автономные действия без подтверждения
- prompt injection из файлов/логов
- эксфильтрация через инструменты (терминал/сеть/интеграции)

## Практический вывод

Если задача **A2+ (мультифайл)**, стандарт должен быть:

- план обязателен
- STOP‑POINT перед изменениями и командами
- проверки обязательны
